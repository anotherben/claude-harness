# Review Lens: SQL — Go

Focused review checklist for Go code interacting with SQL databases. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Go source files (`*.go`) that import `database/sql`, `sqlx`, `pgx`, or any SQL driver package. Any file that constructs or executes SQL queries.

## Checklist

For each changed file matching the applicable pattern, check:

1. Prepared statements only — no `fmt.Sprintf` or string concatenation to build SQL queries (use `$1`/`$2` positional parameters or named parameters)
2. `sql.DB` is a connection pool — never `sql.Open()`/`db.Close()` per query (open once at startup, share the pool)
3. `context.Context` passed to all `QueryContext`, `ExecContext`, `QueryRowContext` calls (not the non-context variants `Query`, `Exec`, `QueryRow`)
4. `rows.Close()` deferred immediately AFTER checking `rows.Err()` is nil — `defer rows.Close()` must appear right after the `Query` call, and `rows.Err()` must be checked after the scan loop
5. Named parameters or positional `$1`/`$2` placeholders — not `?` placeholders (unless using MySQL driver specifically)
6. Transaction rollback in `defer` — `defer tx.Rollback()` immediately after `db.BeginTx()`, with `tx.Commit()` at the end (rollback is a no-op after commit)

## Common Pitfalls

- `fmt.Sprintf("SELECT * FROM users WHERE id = '%s'", id)` — classic SQL injection, use `db.QueryContext(ctx, "SELECT * FROM users WHERE id = $1", id)`
- Opening a new `sql.DB` in every handler function, exhausting database connections in seconds under load
- Using `db.Query()` instead of `db.QueryContext()` — the query cannot be cancelled when the client disconnects
- Forgetting `defer rows.Close()` — connection held open until garbage collection, causing pool exhaustion
- Missing `defer tx.Rollback()` — if any operation between Begin and Commit panics, the transaction hangs until timeout

## Patterns to Enforce

- All SQL must use parameterized queries — `fmt.Sprintf` in any SQL string is an automatic FAIL
- All query and exec calls must use the `Context` variants that accept `context.Context`
- Transactions must have `defer tx.Rollback()` immediately after creation
