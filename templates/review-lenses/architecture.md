# Review Lens: Architecture

Focused review checklist for code architecture and structure. Generated by harness-init from stack-profile.json.

## When This Lens Applies

All changed files. Every PR gets an architecture review pass to catch structural problems early.

## Checklist

For each changed file matching the applicable pattern, check:

1. God file detection — changed file exceeds {{FILE_SIZE_SOFT_LIMIT}} lines (flag for splitting)
2. One write path per table — no duplicate INSERT/UPDATE statements targeting the same table across the codebase
3. No code duplication — no copy-pasted logic across changed files (extract to shared module)
4. Dependency direction — lower layers do not import from higher layers (data layer must not import from routes/controllers)
5. Search before creating — verify no similar function, component, or utility already exists before adding a new one
6. Scope discipline — all changes match the stated task, no drive-by refactors or unrelated fixes

## Common Pitfalls

- Adding a second "create user" function instead of extending the existing one
- Controller importing directly from another controller instead of sharing via a service
- "While I'm here" refactors that bloat the diff and obscure the real change
- Utility files that grow past 800 lines because everything gets dumped there
- Circular dependencies between modules that only surface at runtime

## Patterns to Enforce

- One canonical write path per database table — all writes go through a single service/repository function
- Changes stay within the scope of the task — unrelated improvements go in separate PRs
- New files justify their existence — if a similar file exists, extend it instead
