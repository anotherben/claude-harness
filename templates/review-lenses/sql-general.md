# Review Lens: SQL — General

Focused review checklist for SQL across any database engine. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Any file containing SQL queries or database interactions — ORM calls, raw queries, migration files. Use this lens when no database-specific lens (sql-pg, sql-go) applies.

## Checklist

For each changed file matching the applicable pattern, check:

1. Parameterized queries only — no string concatenation, template literals, or f-strings in SQL statements
2. Tenant scoping — every query that reads or writes tenant data includes a {{TENANT_FIELD}} filter (if {{TENANT_ENABLED}})
3. Migration idempotency — migrations can run twice without error (use IF NOT EXISTS / IF EXISTS guards)
4. No raw user input in queries — all user-supplied values pass through the query parameter mechanism, never interpolated directly

## Common Pitfalls

- ORM methods that accept raw SQL fragments (e.g., `.extra()`, `.raw()`) bypassing parameterization
- Tenant filter present on the main query but missing on subqueries, CTEs, or joined tables
- Migrations that fail on re-run because they assume a clean state
- Dynamic column or table names built from user input — even parameterized queries cannot protect identifiers

## Patterns to Enforce

- All database interactions must use the query library's parameterization mechanism
- Tenant filtering must be verified at the query level, not assumed from upstream middleware
