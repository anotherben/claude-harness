# Review Lens: Tests — JavaScript

Focused review checklist for Jest and Vitest test files. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Test files matching `**/*.test.js`, `**/*.test.ts`, `**/*.test.jsx`, `**/*.test.tsx`, `**/*.spec.js`, `**/*.spec.ts`. Any file in a `__tests__/` directory.

## Checklist

For each changed file matching the applicable pattern, check:

1. Assertions are meaningful — not just `.not.toThrow()` or `.toBeTruthy()` (assert on specific values and shapes)
2. Mocks do not hide the thing being tested — if you mock the database and test database writes, you are testing the mock
3. Test isolation — no shared mutable state between tests (each test sets up and tears down its own state)
4. Edge cases present — null, undefined, empty string, zero, negative numbers, boundary values
5. No snapshot tests as primary assertions — snapshots are supplementary, not the main verification
6. Describe/it names describe BEHAVIOR not implementation (e.g., "rejects expired tokens" not "calls validateToken")
7. AAA structure — Arrange, Act, Assert are clearly separated (setup, execution, verification in distinct blocks)
8. Test file mirrors source file structure — test file location and naming correspond to the source file being tested
9. **Evidence tier matches change scope** — check `.claude/evidence/last-test-run.json` for the `tier` field:
   - If changed files touch database queries, models, or repositories and tier is `mocked` → **FAIL**: "changes touch database layer but only mocked tests ran"
   - If changed files touch API routes or controllers and tier is `mocked` → **FAIL**: "changes touch API layer but only mocked tests ran"
   - If tier is `integration` or `e2e` → evidence is credible for these changes

## Common Pitfalls

- Testing that a mock was called instead of testing the actual outcome — proves wiring, not correctness
- Shared `let` variables mutated across tests, causing order-dependent failures in `--runInBand`
- Snapshot tests that get auto-updated without review, silently accepting regressions
- Missing `await` on async assertions, causing tests to pass without actually running the assertion
- Testing implementation details (internal method calls) instead of observable behavior (return values, side effects)

## Patterns to Enforce

- Every test must have at least one assertion that checks a concrete value, not just truthiness
- Test names must read as behavior specifications — a non-developer should understand what is being verified
- Mocks are scoped to the boundary of the unit under test — external services yes, internal logic no
