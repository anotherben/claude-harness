# Review Lens: API — Python

Focused review checklist for Python web APIs using Django, FastAPI, or Flask. Generated by harness-init from stack-profile.json.

## When This Lens Applies

View, route, and endpoint files in {{SOURCE_DIR}}. Any file that defines HTTP endpoints using Django views/viewsets, FastAPI router functions, or Flask route handlers.

## Checklist

For each changed file matching the applicable pattern, check:

1. Endpoint security — authentication decorators or dependencies are present on all non-public endpoints (`@login_required`, `Depends(get_current_user)`, `@auth.login_required`)
2. Input validation — Pydantic models for FastAPI, serializers for Django REST Framework, WTForms or marshmallow for Flask (no manual request.json parsing without validation)
3. ORM efficiency — `select_related`/`prefetch_related` for Django, `joinedload`/`subqueryload` for SQLAlchemy (no N+1 queries in loops)
4. Response serialization — sensitive fields excluded from response (passwords, tokens, internal IDs not meant for clients)
5. Error responses follow a consistent format — same error shape across all endpoints (e.g., `{"detail": "message", "code": "ERROR_CODE"}`)
6. No bare `except:` clauses — all exception handling specifies the exception type (at minimum `except Exception:`)

## Common Pitfalls

- Django queryset evaluated in a loop without `prefetch_related`, generating one query per iteration
- FastAPI endpoint missing `Depends(get_current_user)` — endpoint is publicly accessible
- Bare `except:` catching `SystemExit` and `KeyboardInterrupt`, making the process unkillable
- Returning the full ORM model including `password_hash` or `api_key` fields in the API response
- Flask route missing `methods=['POST']`, defaulting to GET-only and rejecting form submissions

## Patterns to Enforce

- Every endpoint must have explicit authentication unless it is documented as public
- All user input must pass through a validation layer before reaching business logic
- ORM queries in loops must use prefetching — verify with query logging if uncertain
