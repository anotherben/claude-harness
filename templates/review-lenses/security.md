# Review Lens: Security

Focused review checklist for application security. Generated by harness-init from stack-profile.json.

## When This Lens Applies

All changed files. Every PR gets a security review pass regardless of stack or file type.

## Checklist

For each changed file matching the applicable pattern, check:

1. No string concatenation or template literals in SQL queries or shell commands (injection)
2. All non-public endpoints have authentication middleware or decorators (broken auth)
3. No secrets, API keys, passwords, or tokens hardcoded in source (sensitive data exposure)
4. No PII or credentials written to logs (sensitive data logging)
5. User input is sanitized before rendering in HTML/JSX (XSS)
6. CORS configuration does not use wildcard `*` in production (CORS misconfiguration)
7. No use of `eval()`, `Function()` constructor, or `vm.runInNewContext()` with user input (code injection)
8. No shell command execution with unsanitized user input — no `child_process.exec(userInput)` (command injection)
9. Rate limiting present on authentication and password-reset endpoints (brute force)

## Common Pitfalls

- Template literals in SQL: `` `SELECT * FROM users WHERE id = ${id}` `` — use parameterized queries instead
- Logging request bodies that contain passwords or tokens
- CORS set to `*` during development and never changed for production
- Secrets in config files committed to repo instead of environment variables
- Using `dangerouslySetInnerHTML` or equivalent without sanitization

## Patterns to Enforce

- All SQL must use parameterized queries with positional or named placeholders
- Secrets must come from environment variables, never from source files
- Auth middleware must be applied at the router level, not per-handler
