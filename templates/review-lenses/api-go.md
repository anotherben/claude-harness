# Review Lens: API — Go

Focused review checklist for Go HTTP APIs and services. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Go source files (`*.go`) that define HTTP handlers, middleware, services, or any code with I/O operations. Excludes test files (those use the test-go lens if present).

## Checklist

For each changed file matching the applicable pattern, check:

1. Error handling — no `_ = fn()` ignoring returned errors, no `err` shadowing with `:=` in nested scopes (use `=` when `err` is already declared)
2. Goroutine lifecycle — every goroutine has a termination path, context cancellation is propagated, no goroutine leaks
3. `context.Context` as first parameter for all functions that perform I/O (database calls, HTTP requests, file operations)
4. Interface satisfaction checked at compile time — `var _ Interface = (*Struct)(nil)` for types that must implement an interface
5. `defer` for cleanup — response bodies (`resp.Body.Close()`), file handles, mutex unlocks are deferred immediately after acquisition
6. No `panic` in library code — panics are reserved for truly unrecoverable states in `main`; libraries return errors

## Common Pitfalls

- `err` shadowed by `:=` in an inner `if` block, causing the outer `err` to remain nil and hiding the real error
- Goroutine started in a handler without context — lives forever if the parent request is cancelled
- Missing `defer resp.Body.Close()` on HTTP client responses, causing connection pool exhaustion
- `panic` in a library function that crashes the entire server instead of returning an error to the caller
- `context.Background()` used where a request context should be propagated, breaking cancellation and timeouts

## Patterns to Enforce

- Every returned error must be checked — no blank identifier for error returns
- Goroutines must accept a context and select on `ctx.Done()` for graceful shutdown
- Resource cleanup must use `defer` immediately after the resource is acquired
