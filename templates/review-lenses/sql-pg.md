# Review Lens: SQL — PostgreSQL

Focused review checklist for PostgreSQL queries and migrations. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Any file containing SQL queries targeting PostgreSQL — including route handlers, services, repositories, and migration files in {{MIGRATION_DIR}}.

## Checklist

For each changed file matching the applicable pattern, check:

1. Parameterized queries ONLY — zero template literals or string concatenation in SQL (use `$1`, `$2` placeholders)
2. {{TENANT_FIELD}} scoping on every SELECT, INSERT, UPDATE, and DELETE (if {{TENANT_ENABLED}}) — no query may omit tenant filtering
3. JOIN type safety — UUID columns and integer columns must be cast to text before comparison (e.g., `suppliers.id::text = products.supplier_id::text`)
4. IF NOT EXISTS guards on all CREATE TABLE, CREATE INDEX, and ALTER TABLE ADD COLUMN statements
5. TIMESTAMPTZ used instead of TIMESTAMP for all date/time columns (timezone-aware storage)
6. New foreign keys have corresponding indexes on the referencing column
7. Migrations are idempotent — running the same migration twice must not produce errors
8. No DROP TABLE or DROP COLUMN without IF EXISTS guard

## Common Pitfalls

- Joining `suppliers.id` (UUID) to `products.supplier_id` (integer) without casting — silently returns zero rows
- Using TIMESTAMP instead of TIMESTAMPTZ, causing timezone bugs when the server moves regions
- Missing {{TENANT_FIELD}} in a subquery or CTE while the outer query has it — data leak across tenants
- CREATE INDEX without IF NOT EXISTS failing on re-deploy because the index already exists
- Forgetting to add an index on a new foreign key column, causing slow JOINs on large tables

## Patterns to Enforce

- Every SQL query that touches tenant data must include `WHERE {{TENANT_FIELD}} = $N`
- All DDL statements must be safe to run repeatedly (IF NOT EXISTS / IF EXISTS)
- No raw string interpolation in any SQL — parameterized queries are the only acceptable pattern
