# Review Lens: Tests — Python

Focused review checklist for pytest test files. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Test files matching `**/test_*.py`, `**/*_test.py`. Any file in a `tests/` directory using pytest conventions.

## Checklist

For each changed file matching the applicable pattern, check:

1. Assertions use pytest idioms — plain `assert x == y`, not `self.assertEqual(x, y)` (unless unittest is the project standard)
2. Fixtures used for setup — not `setUp`/`tearDown` methods (prefer pytest fixtures with appropriate scope)
3. `@pytest.mark.parametrize` for multiple similar test cases — no copy-pasted tests with different inputs
4. Mocks scoped correctly — not leaking between tests (use `monkeypatch` or properly scoped `mock.patch`)
5. Test names are descriptive — `test_should_reject_invalid_email` not `test_1` or `test_email`
6. Edge cases covered — None, empty string, empty list, zero, negative values, boundary conditions

## Common Pitfalls

- Module-scoped fixtures that mutate state, causing test-order-dependent failures
- Using `mock.patch` as a decorator but forgetting that it adds a parameter to the test function
- Parametrize IDs missing, making failure output unreadable (`FAILED test_x[param0]` instead of a meaningful label)
- Asserting on mutable return values that change after the assertion (e.g., asserting on a dict that gets modified later)
- Forgetting to mark async tests with `@pytest.mark.asyncio`

## Patterns to Enforce

- Every test function name must describe the expected behavior and condition
- Fixtures must have the narrowest possible scope — function scope by default, broader only with justification
- Parametrize must include `ids` for readable test output
