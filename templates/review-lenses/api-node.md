# Review Lens: API — Node.js

Focused review checklist for Node.js Express/Fastify APIs. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Route files, controllers, and middleware in {{SOURCE_DIR}}. Any file that defines HTTP endpoints, request handlers, or middleware functions.

## Checklist

For each changed file matching the applicable pattern, check:

1. Route ordering — public and webhook routes are registered BEFORE {{AUTH_MIDDLEWARE}} middleware (auth must not block public endpoints)
2. Error handling — no empty catch blocks, no swallowed errors (every catch logs or re-throws)
3. Async/await — every async call is awaited, or intentionally fire-and-forget with an explicit comment explaining why
4. Response shape consistency — all endpoints in the same router return the same top-level shape (e.g., `{ data, error }`)
5. Middleware chain — new routes have appropriate middleware applied (auth, validation, rate limiting)
6. No double response — no `res.send()` after `res.json()`, no missing `return` before response in conditional branches
7. Status codes correct — 201 for resource creation, 204 for deletion, 404 for not found (not 200 for everything)

## Common Pitfalls

- Placing a new public route after the auth middleware mount point, making it inaccessible without a token
- Forgetting `return` after sending a response in an `if` block, causing "headers already sent" errors
- Using `async` on a handler but not awaiting a database call, causing unhandled promise rejections
- Returning 200 with an error message in the body instead of using proper 4xx/5xx status codes
- Missing `next(err)` in middleware, causing requests to hang

## Patterns to Enforce

- Every route handler must have error handling — either try/catch or an error-handling middleware
- Response status codes must match the HTTP semantics of the operation
- New routes must be added in the correct position relative to {{AUTH_MIDDLEWARE}}
