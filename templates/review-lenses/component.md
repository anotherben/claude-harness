# Review Lens: Component Structure

Focused review checklist for component architecture and responsibility. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Component files in {{FRONTEND_DIR}}. Any file that exports a React component, Vue component, or Svelte component. Focus on structural concerns rather than framework-specific API usage.

## Checklist

For each changed file matching the applicable pattern, check:

1. Component is under {{FILE_SIZE_SOFT_LIMIT}} lines — if over, flag for splitting into smaller components
2. Props are typed — PropTypes definitions, TypeScript interfaces, or JSDoc type annotations present
3. Separation of concerns — data fetching logic lives in hooks or services, not inline in the render function or component body
4. Existing components reused — search the codebase for similar components before creating a new one (no duplicate Button, Modal, Table)
5. No business logic in component body — complex calculations, data transformations, and validation extracted to custom hooks or utility functions
6. Single responsibility — the component does one thing (renders a form, displays a list, handles a modal) not multiple unrelated things

## Common Pitfalls

- A "page" component that fetches data, transforms it, handles form state, and renders everything — split into container and presentation
- Creating a new `ConfirmDialog` when one already exists in the shared components directory
- Inline data transformation in JSX: `{items.filter(x => x.active).map(x => ({...x, label: x.name.toUpperCase()})).map(x => <Item key={x.id} {...x} />)}` — extract to a variable or hook
- Props with no type definitions, making the component API undiscoverable
- Components over 400 lines that "just need one more feature" instead of being decomposed

## Patterns to Enforce

- Components must have typed props — no untyped component interfaces
- Data fetching and business logic must live outside the component render path
- Before creating a new shared component, verify no equivalent exists in the codebase
