# Review Lens: Frontend — React

Focused review checklist for React components with Vite or Next.js. Generated by harness-init from stack-profile.json.

## When This Lens Applies

Component and page files matching {{FRONTEND_DIR}}/**/*.jsx, {{FRONTEND_DIR}}/**/*.tsx. Any file that imports from `react` and returns JSX.

## Checklist

For each changed file matching the applicable pattern, check:

1. Hook rules — no hooks called inside conditionals, loops, or callback functions (hooks must be at the top level of the component or custom hook)
2. Render performance — no inline object/array literals as props (`style={{color: 'red'}}` creates a new object every render), no anonymous functions in JSX for items in large lists
3. State management — state lives at the lowest common ancestor, Context is not used for high-frequency updates (use state management library or lifting state instead)
4. Accessibility — form inputs have associated labels, images have alt text, interactive elements have appropriate ARIA roles
5. No `window.confirm()` or `window.alert()` — use the project dialog service for all confirmation and alert dialogs
6. useEffect dependencies complete — no missing dependencies in the dependency array (ESLint exhaustive-deps rule must pass)
7. Cleanup in useEffect — return a cleanup function for subscriptions, timers, event listeners, and abort controllers

## Common Pitfalls

- Missing cleanup in useEffect causing "setState on unmounted component" warnings and memory leaks
- Using Context for theme or locale (low frequency) is fine; using it for form state (high frequency) causes unnecessary re-renders
- Inline `onClick={() => handleClick(id)}` in a list of 1000 items creates 1000 new function references on every render
- `window.confirm()` blocks the main thread and cannot be styled — always use the project dialog service
- Missing key prop on list items, or using array index as key when items can reorder

## Patterns to Enforce

- All side effects with external subscriptions must return a cleanup function from useEffect
- Dialog interactions must use the project dialog service, never native browser dialogs
- Props that are objects or arrays must be stable references (useMemo, useState, or module-level constants)
